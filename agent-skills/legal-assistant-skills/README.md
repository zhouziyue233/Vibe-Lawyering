## 📋 原版产品功能（Contract Review Skill）

### 四层审查模型

1. **Layer 0: 主体核验**
   - 提取合同相对方名称
   - 核验工商登记状态、统一社会信用代码
   - 标记主体异常风险

2. **Layer 1: 基础文本审查**
   - 数字、日期、条款准确性
   - 编号一致性检查
   - 格式和标点质量

3. **Layer 2: 商务条款审查**
   - 服务范围、交付物、规格
   - 价格和付款计划
   - 权利义务和履约保障

4. **Layer 3: 法律条款审查**
   - 生效、期限、终止条件
   - 违约责任、争议解决
   - 保密、不可抗力、知识产权

### 主要特点

- ✅ **仅添加评论，不改原文** - 保持合同原始文本完整
- ✅ **评论结构标准化** - 问题类型、风险原因、修订建议
- ✅ **风险等级编码** - 通过 reviewer 名称标识 🔴🟡🔵
- ✅ **多语言支持** - 自动检测合同语言，输出对应语言结果

### 标准输出

| 输出文件 | 格式 | 说明 |
|---------|------|------|
| `{合同名}_审核版.docx` | DOCX | 带批注的审核版合同 |
| `合同概要.docx` | DOCX | 结构化合同摘要 |
| `综合审核意见.docx` | DOCX | 给业务团队的审核意见 |
| `业务流程图.mmd` | Mermaid | 业务流程图源码 |
| `审核报告.txt` | TXT | 审核统计报告 |

## ⚡ 30秒快速体验

### 前置准备

```bash
# 1. 安装Python依赖（只需一次）
pip3 install requests

# 2. 配置企查查MCP Key（从 https://agent.qcc.com 申请）
# 方式一：临时设置（当前终端有效）
export QCC_MCP_API_KEY="your_key_here"

# 方式二：永久设置（推荐）
# 添加到shell配置文件，这样无论从哪启动Claude Code都能读取到
echo 'export QCC_MCP_API_KEY="your_key_here"' >> ~/.zshrc
source ~/.zshrc
# 如果是Bash用户，请使用 ~/.bashrc
```

**注意**：如果使用方式一（临时设置），必须在同一终端窗口中启动Claude Code，否则环境变量不会生效。强烈建议使用方式二（永久设置）。

### 一键安装

```bash
# 3. 运行安装脚本（自动配置 MCP）
bash <(curl -sL https://raw.githubusercontent.com/duhu2000/legal-assistant-skills/main/install_qcc_mcp.sh)
```

### 重要：重启 Claude Code

⚠️ **必须完全退出并重新启动 Claude Code**，否则 MCP 配置不会生效。

```bash
# 4. 重启 Claude Code
# - 完全退出 Claude Code（Cmd+Q 或关闭终端）
# - 重新启动 Claude Code

# 5. 验证 MCP 配置
# 你应该能看到可用的 MCP 工具：qcc-company, qcc-risk, qcc-ipr, qcc-operation
```

### 开始使用


**Done!** 系统自动完成：
- ✅ 提取甲方/乙方等合同主体

## 💡 典型应用场景（法务/律师专用）

### 场景1：合同签署前主体核验

**业务背景**：法务审核合同时，需要确认合同对方主体是否合法存续、是否存在经营风险。

**传统方式**：
- 手工登录国家企业信用信息公示系统
- 逐个查询合同相对方
- 人工比对营业执照信息
- 耗时：15-30分钟/合同

**企查查MCP智能审核**：
```
AI: 读取《产品服务合同》
    ↓
AI: 提取甲方「绿地控股集团有限公司」、乙方「企查查科技股份有限公司」
    ↓
MCP: 自动核验双方主体
     ├─ 绿地控股：✅ 存续状态，注册资本XXX万，法人XXX
     └─ 企查查科技：✅ 存续状态，统一社会信用代码9132XXXXXXXX
    ↓
AI: 生成主体核验批注
     ├─ 批注1: ✅ 主体核验通过 - 绿地控股集团工商登记有效
     └─ 批注2: ✅ 主体核验通过 - 企查查科技工商登记有效
```

**价值**：3秒自动完成主体核验，无需手动查询，直接在合同中标注核验结果。

---

### 场景2：风险主体识别与预警

**业务背景**：合同相对方存在经营异常、失信记录等风险，需要及时识别并提示业务终止签约。

**典型场景**：
```
AI: 检测到合同甲方「XXX科技有限公司」
    ↓
MCP: 实时核验发现风险
     ├─ ⚠️ 经营异常：2024-03-15被列入异常名录
     ├─ ⚠️ 股权冻结：控股股东股权被司法冻结
     └─ ⚠️ 受限高消费：法定代表人被限高
    ↓
AI: 在合同中自动标注高风险批注
     ├─ 🔴 高风险-经营异常
     │   建议：要求对方提供移出异常名录证明，或更换签约主体
     ├─ 🔴 高风险-股权冻结
     │   建议：核实股权冻结原因，评估履约能力
     └─ 🔴 高风险-法定代表人限高
         建议：谨慎签约，建议要求提供履约担保
```

**价值**：在签约前主动发现风险，避免与失信企业签约，降低合同履行风险。

### 场景3：批量合同主体筛查

**业务背景**：律所/企业法务部需要对大量存量合同进行风险排查，识别合作方风险变化。

**工作流程**：
```

    ↓
MCP: 自动提取100个合同相对方
    ↓

     ├─ 经营异常：3家

     └─ 股权冻结：1家
    ↓
AI: 生成风险排查报告
     ├─ 《高风险合同清单》- 需立即处理
     ├─ 《中风险合同清单》- 需关注
     └─ 《合同主体风险分析报告》- 含风险分布统计
```

**价值**：批量筛查存量合同，及时发现合作方风险变化，为主动风险管理提供数据支持。

## 🎓 企查查MCP工作原理：Agent-Native设计

### Step 1: 合同解析
- 提取合同文本
- 识别甲方/乙方/买方/卖方等主体

### Step 2: 企业核验（MCP实体锚定）
```
传统API: "华为技术有限公司" → 可能返回多个相似名称
MCP方案: "华为技术有限公司"
         → 强制匹配统一社会信用代码914403001038138210
         → 返回"实体锚定确认：唯一有效主体"
```

### Step 3: 风险扫描（MCP语义防御）
```
传统API: 无失信记录 → 返回 []
         → LLM困惑："真没有？还是查失败了？"
MCP方案: 无失信记录 → 返回 "核验通过：未发现失信信息"
         → LLM确定："合规状态安全，可继续执行"
```

### Step 4: 上下文脱水（MCP Token优化）
```
传统API: 返回30000字全量数据 → Token爆炸
MCP方案: 智能摘要+按需索取 → 仅1000字关键信息
         → 成本降低96%，决策效率提升
```

### Step 5: 批注生成
- 核验批注：企业信息确认（含实体锚定标识）
- 风险批注：风险提示+法律建议
- 审计留痕：所有核验可追溯

## 🌟 为什么选择企查查MCP原生方案

基于**Agent-Native**理念打造的企业数据基座，专为LLM优化：

| 核心能力 | 传统API痛点 | 企查查MCP方案优势 |
|---------|-------------|-------------|
| **零幻觉核验** | 名称匹配出错，张冠李戴（如错认乐视网） | 强制验证统一社会信用代码，物理切断名称幻觉 |
| **Token优化** | 返回数万字JSON，瞬间撑爆上下文 | 智能摘要+按需索取，Token开销直降96% |
| **确定性决策** | 空数据返回[]，LLM陷入自我怀疑 | 强语义状态码（如"核验通过"），颁发确定性合规通行证 |
| **合规可审计** | 无核验留痕，无法追溯 | 全链路留痕，满足法务审计要求 |
| **LLM原生** | 原始JSON需解析转换 | 自然语言输出，语义即用，零集成成本 |
